# 多源同步 功能设计

文件：`src/services/sync.ts`

## 现状

- SyncService 已实现三源发现（Gateway / Disk / Config）
- 支持 dry run 预览和实际同步
- 内容哈希对比实现增量更新
- 消失的 Agent 自动标记 offline
- 全程审计日志

---

## 一、发现器

### 1.1 优先级

```
Gateway（最高） > Disk > Config（最低）
```

同一 `id` 的 Agent 出现在多个源时，高优先级源覆盖低优先级。

### 1.2 Gateway 发现

```
条件：client.connected === true
调用：client.request("agents.list", {})
映射：result[] → DiscoveredAgent[]
失败：静默跳过（warn 日志），返回空数组
```

字段映射：

| 网关字段 | → | 本地字段 |
|----------|---|---------|
| id / name | → | id |
| name | → | name |
| connectionId | → | sourceRef |
| status | → | status |
| model | → | model |
| workspace | → | workspace |
| SHA-256(JSON) | → | contentHash |

### 1.3 磁盘发现

```
目录：AGENTS_DIR（默认 ~/.openclaw/agents）
扫描：遍历一级子目录
每个目录读取：
  - identity.json  → 基础信息（id / name / emoji / role / model / workspace / config）
  - soul.md        → Soul 内容
失败：目录不存在或不可读时静默跳过
```

目录结构示例：

```
~/.openclaw/agents/
├── alice/
│   ├── identity.json    { "name": "Alice", "model": "gpt-4", ... }
│   └── soul.md          "你是一个代码审查专家..."
├── bob/
│   └── identity.json
└── charlie/
    └── soul.md
```

- 没有 `identity.json` 时，使用目录名作为 id 和 name
- 没有 `soul.md` 时，soul 为 null
- contentHash = SHA-256(JSON.stringify({ identity, soul }))

### 1.4 配置文件发现

```
路径：OPENCLAW_CONFIG（默认 ~/.openclaw/openclaw.json）
解析：JSON → config.agents.list[]
失败：文件不存在时静默跳过
```

> **Bug 修正**：当前代码把 `config.agents` 当 flat array 解析，但真实 `openclaw.json` 中 agents 是嵌套结构（`agents.defaults` + `agents.list`），需要修正。

实际配置文件格式：

```json
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "provider/model-id",
        "fallbacks": ["provider2/model-id"]
      },
      "workspace": "~/.openclaw/workspace",
      "maxConcurrent": 4
    },
    "list": [
      {
        "id": "main",
        "name": "main",
        "default": true,
        "workspace": "~/.openclaw/workspace"
      },
      {
        "id": "ceo",
        "name": "项目总监",
        "workspace": "~/.openclaw/workspace-ceo",
        "subagents": { "allowAgents": ["qa", "backend"] }
      }
    ]
  }
}
```

解析规则：

- Agent 列表：`config.agents.list[]`（非 `config.agents[]`）
- 每个 agent 的 `id` 是必须的，`name` 缺失时用 `id`
- `defaults` 字段提供继承值，agent 自身字段覆盖 defaults
- `model` 字段是对象（`{ primary, fallbacks }`），同步时取 `model.primary` 存入 DB
- `workspace` 支持 `~` 开头的路径，需 resolvePath

字段映射：

| 配置文件字段 | → | DB 字段 |
|-------------|---|---------|
| id | → | id |
| name / id | → | name |
| workspace / defaults.workspace | → | workspace |
| model.primary / defaults.model.primary | → | model（取 provider/model-id 字符串） |
| — | → | source = "config" |
| 配置文件路径 | → | sourceRef |
| SHA-256(JSON) | → | contentHash |

## 二、合并引擎

### 2.1 流程

```
1. 并行调用三个发现器（Promise.all）
   ↓
2. 按优先级去重合并（Map，后写入覆盖前写入）
   config → disk → gateway
   ↓
3. 对比现有数据库记录
   ↓
4a. dryRun=true  → 返回预览（created / updated 列表）
4b. dryRun=false → 执行实际写入
   ↓
5. 标记消失的 Agent 为 offline
   ↓
6. 触发 agent.synced 事件
```

### 2.2 增量判断

通过 `contentHash` 对比决定是否需要更新：

- 新 Agent（DB 中不存在）→ INSERT
- contentHash 变化 → UPDATE（仅更新变化字段）
- contentHash 相同 → 跳过
- DB 中存在但发现器未返回 → 标记 offline（仅非 local source）

### 2.3 返回值

```typescript
interface SyncResult {
  created: string[]   // 新增的 agent id 列表
  updated: string[]   // 更新的 agent id 列表
  offlined: string[]  // 标记为 offline 的 agent id 列表
}
```

## 三、API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/agents/sync` | 执行同步，返回 SyncResult |
| POST | `/api/agents/sync/preview` | 预览同步（dryRun=true），不实际写入 |

## 四、优化清单

| # | 问题 | 方案 |
|---|------|------|
| 1 | `discoverFromConfig()` 解析路径错误 | 改为读取 `config.agents.list`，合并 `config.agents.defaults` |
| 2 | `model` 字段是对象不是字符串 | 取 `model.primary`（如 `"minimax-cpa/MiniMax-M2.5-highspeed"`） |
| 3 | 缺少 Skill 同步 | 从 Gateway RPC 同步 Skill 数据（`skills.list`） |
| 4 | dryRun 不检测 offlined | 预览时也应显示即将被标记 offline 的 agent |
| 5 | 复用 `gateway/config.ts` | 配置文件读取逻辑抽到 `gateway/config.ts`，sync 和 gateway 初始化共用 |
